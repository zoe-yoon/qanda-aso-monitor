## Executive Summary

ASO Keyword Monitor는 콴다 ASO 팀(2-3명)이 사용하는 내부 전용 웹 대시보드다. 핵심 UX 문제는 "데이터 분산으로 인한 인지 부하와 수작업 반복"이며, 디자인 목표는 이를 단일 통합 인터페이스로 수렴시키는 것이다.

**핵심 디자인 시사점 3가지:**

1. **데이터 신뢰도 가시화**: 자동 수집된 데이터는 항상 수집 시각, 출처(센서타워/ASC), 수집 성공/실패 상태를 함께 표시해야 한다. 사용자가 "이 데이터가 오늘 것인가?"를 묻지 않아야 한다.

2. **주력/보조 분류 체계가 정보 구조의 중심**: 콴다 팀 고유의 방어/확장 키워드 분류가 내비게이션, 탭, 필터, 제안 기능 전반의 핵심 IA(Information Architecture) 축이다.

3. **3개 사용자 역할의 다른 진입 목적**: 민지(변동 감지 즉시 확인), 준호(키워드 발굴 탐색), 재현(성과 요약 열람)이 같은 대시보드를 다른 목적으로 진입한다. 각 역할의 최우선 정보가 초기 화면에 드러나야 한다.

## Section 1. Personas

### Persona P-01: 민지 (ASO 매니저)

```
persona_id     : P-01
name           : 민지
role           : ASO 매니저 (Primary User)
age_range      : 20대 후반
experience     : 모바일 앱 마케팅 3년차

goals:
  - 키워드 랭킹 변동을 당일, 가능하면 실시간에 가깝게 감지
  - 주력(방어) 키워드 Top 10의 성과를 매일 아침 빠르게 파악
  - ASO 점수 81% → 85% 달성 진척도 추적
  - 데이터 수집에 쓰는 시간을 전략적 분석으로 전환

pain_points:
  - 매일 센서타워 + App Store Connect + Google Play Console 3곳을 열어
    수기로 스프레드시트에 입력하는 데 30분~1시간 소요
  - 랭킹이 전날 하락했어도 다음날 수기 확인 전까지 인지 불가
  - 교차 분석(랭킹 하락 → CVR 하락 연결) 불가능

tech_savviness : High
  - Google Sheets 고급 사용자 (수식, 조건부 서식 직접 작성)
  - 센서타워, App Store Connect 대시보드 능숙하게 탐색
  - Slack 알림을 통한 업무 처리에 익숙

usage_context:
  - 환경: 데스크탑 웹 브라우저 (Chrome), 사무실 근무
  - 빈도: 매일 오전 첫 업무로 대시보드 확인 (Daily Active User)
  - 보조: Slack 알림 수신 시 모바일에서도 확인할 가능성 있음

key_scenarios:
  1. [데일리 모닝 체크] 오전 출근 후 대시보드를 열어 어제 대비
     키워드 랭킹 변동이 큰 키워드를 즉시 파악한다.
  2. [이상 감지 대응] Slack에서 "수학 키워드 -5위" 알림을 받고
     대시보드에서 해당 키워드의 랭킹 추이, CVR 변화를 교차 확인한다.
  3. [주간 성과 리포트] 매주 월요일, 지난주 주력 키워드 성과를
     요약하여 PM(재현)에게 보고한다.

accessibility_needs: 특별한 접근성 니즈 없음
```

### Persona P-02: 준호 (ASO 실무자)

```
persona_id     : P-02
name           : 준호
role           : ASO 실무자 / 키워드 리서치 담당 (Primary User)
age_range      : 20대 중반
experience     : 마케팅 1-2년차

goals:
  - 트래픽 3.0+ 기준을 충족하는 신규 보조(확장) 키워드 후보 발굴
  - 말해보카, 스픽, 듀오링고, 찰칵 등 경쟁사 키워드 랭킹 비교
  - 데이터 기반으로 키워드를 선정하여 상급자에게 근거 제시

pain_points:
  - 키워드 선정이 경험과 직감에 의존, 체계적 기준 부재
  - 경쟁사가 어떤 키워드에서 높은 랭킹을 차지하는지 파악하기 어려움
  - 발굴한 키워드 후보를 주력/보조로 분류하는 기준이 불명확

tech_savviness : Medium-High
  - Google Sheets 기본~중급 사용자
  - 센서타워 사용 경험 있으나 고급 기능 활용은 제한적
  - 데이터 분석보다 콘텐츠/키워드 리서치에 강점

usage_context:
  - 환경: 데스크탑 웹 브라우저, 사무실 근무
  - 빈도: 키워드 리서치 주기에 따라 주 2-3회 집중 사용
  - 탐색형 사용 패턴 (여러 탭/필터 전환하며 비교)

key_scenarios:
  1. [신규 키워드 발굴] 키워드 제안 탭에서 트래픽 기준 필터를 적용하고,
     제안된 키워드 후보를 검토하여 보조 키워드 목록에 추가한다.
  2. [경쟁사 벤치마크] 특정 키워드에서 콴다와 경쟁사(말해보카, 스픽)의
     랭킹 순위를 비교하여 경쟁 강도를 평가한다.
  3. [키워드 분류 검토] 현재 주력/보조로 분류된 키워드 목록을 검토하고,
     성과가 낮은 키워드를 제거하거나 재분류한다.

accessibility_needs: 특별한 접근성 니즈 없음
```

### Persona P-03: 재현 (PM / 팀 리더)

```
persona_id     : P-03
name           : 재현
role           : PM / ASO Epic 오너 (Secondary User)
age_range      : 30대 초반
experience     : 제품 조직 리더

goals:
  - ASO 성과(랭킹, 다운로드, CVR)를 담당자를 통하지 않고 직접 확인
  - CAC 10% 절감 목표 달성을 위한 Organic 유입 성과 추적
  - 주간/월간 리포트를 자동으로 수령하여 팀 리뷰에 활용

pain_points:
  - 성과 확인이 필요할 때마다 민지/준호에게 요청해야 함
  - 요약 지표만 필요하지만 Raw 데이터 스프레드시트에 접근해야 함
  - 진척도(ASO 점수 81%→85%)를 한눈에 볼 수 없음

tech_savviness : Medium
  - Google Sheets 기본 수준 열람 (편집보다 조회 중심)
  - 복잡한 필터/수식 사용하지 않음
  - Slack, 이메일로 정기 수령하는 것을 선호

usage_context:
  - 환경: 데스크탑 웹 브라우저, 사무실/재택 혼합
  - 빈도: 주 2-3회 (주간 리포트 수령 시 + 필요 시 직접 확인)
  - 소비형 사용 패턴 (편집 없이 요약 지표 열람)

key_scenarios:
  1. [성과 현황 자가 조회] 민지에게 요청하지 않고 직접 대시보드에서
     이번 주 주력 키워드 랭킹 요약과 다운로드 추이를 확인한다.
  2. [주간 리포트 검토] 자동 생성된 주간 요약 리포트(Slack 또는 웹)에서
     전주 대비 성과 변화와 특이 사항을 확인한다.
  3. [목표 달성 추적] ASO 점수 현황(81%→목표 85%)과 CAC 관련 지표를
     한 화면에서 확인한다.

accessibility_needs: 정보 밀도가 낮은 요약 뷰 선호 (복잡한 표/필터 기피)
```

## Section 2. User Journeys

### Journey J-01: 데일리 모닝 키워드 체크 (P-01 민지)

```
journey_id   : J-01
persona      : P-01 (민지, ASO 매니저)
goal         : 오전 출근 직후 5분 안에 어제 대비 주요 키워드 변동 파악
trigger      : 매일 오전 출근 또는 Slack 데일리 알림 수신
priority     : Critical (Daily Active Use Case)

touchpoints:
  1. [진입] 웹 대시보드 URL 접속 → 통합 Overview 화면 로딩
  2. [스캔] 주력(방어) 키워드 섹션에서 전일 대비 ±변동 하이라이트 스캔
  3. [집중] 빨간색(하락) 하이라이트된 키워드 행 클릭 → 상세 추이 확인
  4. [교차 확인] 해당 키워드의 다운로드, CVR 변화 연관 데이터 확인
  5. [액션 결정] 대응이 필요한 경우 메모/태그 추가 또는 팀 공유

emotional_arc:
  진입(기대/불안) → 스캔(집중) → 이상 없음(안도)/이상 발견(긴장)
  → 교차 확인(분석적 집중) → 액션 결정(완료감 또는 팀과 공유)

key_pain_points:
  - 현재: 대시보드 로딩 후 어디서 변동을 확인해야 할지 불명확
  - 현재: 하락한 키워드를 발견해도 원인 파악을 위해 다른 탭으로 이동해야 함
  - 현재: 어제 데이터가 정상 수집됐는지 확인할 방법이 없음

opportunities:
  - 대시보드 상단에 "오늘 주목할 변동" 요약 카드 배치
  - 하락/상승 키워드를 별도 섹션으로 분리하여 즉시 식별 가능하게
  - 데이터 수집 상태(마지막 업데이트 시각) 항상 표시
  - 하락 키워드 클릭 시 해당 키워드의 랭킹+CVR+다운로드 인라인 표시
```

### Journey J-02: 신규 키워드 발굴 및 분류 (P-02 준호)

```
journey_id   : J-02
persona      : P-02 (준호, ASO 실무자)
goal         : 트래픽 기준을 충족하는 새 보조(확장) 키워드 후보 3개 이상 발굴
trigger      : 정기 키워드 리뷰 주기 도래 또는 민지의 키워드 보강 요청
priority     : High (Core Feature Use Case)

touchpoints:
  1. [진입] 키워드 제안 탭 접속
  2. [필터 설정] 트래픽 점수 3.0+ 필터 적용, 현재 미등록 키워드만 표시
  3. [탐색] 제안 목록 스캔 - 키워드명, 트래픽 점수, 현재 랭킹 확인
  4. [경쟁사 비교] 관심 키워드에서 경쟁사 랭킹 열 확인
  5. [채택 결정] 후보 키워드를 "보조 키워드 목록에 추가" 액션 실행
  6. [검증] 보조 키워드 탭으로 이동하여 추가된 키워드 확인

emotional_arc:
  진입(탐색 기대) → 필터/탐색(몰입) → 후보 발견(흥미)
  → 경쟁사 비교(비교 분석) → 채택(성취감) → 검증(완료 확인)

key_pain_points:
  - 현재: 키워드 제안 기준(트래픽 3.0+)이 화면에 명시되지 않아 기준 인지 어려움
  - 현재: 제안 키워드와 현재 등록 키워드 간 중복 여부 파악 필요
  - 현재: 경쟁사와 콴다 랭킹을 같은 행에서 비교할 수 없음

opportunities:
  - 필터 패널에 "부수 키워드 기준" (트래픽 3.0+, ai 관련 등) 사전 설정 프리셋 제공
  - 이미 모니터링 중인 키워드는 목록에서 시각적으로 구분 표시
  - 제안 키워드 행에서 콴다 vs 경쟁사 순위 인라인 비교 컬럼 제공
  - 채택 버튼 클릭 시 주력/보조 분류 선택 모달 제공
```

### Journey J-03: 이상 감지 및 긴급 대응 (P-01 민지)

```
journey_id   : J-03
persona      : P-01 (민지, ASO 매니저)
goal         : Slack 알림 수신 후 5분 내 상황 파악 및 대응 방향 결정
trigger      : Slack 알림 "키워드 '수학' 랭킹 -5위 변동 감지"
priority     : High (P1 Feature - Slack Alert)

touchpoints:
  1. [알림 수신] Slack 채널에서 랭킹 급변동 알림 확인
  2. [빠른 진입] 알림 내 딥링크로 해당 키워드 상세 화면 바로 접근
  3. [변동 확인] 최근 7일 랭킹 추이 차트에서 하락 시점 확인
  4. [원인 분석] 같은 화면에서 CVR, 다운로드 데이터 동시 확인
  5. [경쟁사 비교] 경쟁사 랭킹 변화로 전체 시장 변동인지 콴다 특이사항인지 구분
  6. [대응 기록] 분석 내용을 팀 채널에 공유 또는 대시보드 내 메모

emotional_arc:
  알림 수신(긴장/즉각 반응) → 진입(집중) → 변동 확인(우려)
  → 원인 분석(분석적 긴장) → 경쟁사 비교(맥락 파악)
  → 대응 결정(안도 또는 추가 조치 필요)

key_pain_points:
  - 알림에 딥링크가 없으면 대시보드에서 해당 키워드를 다시 찾아야 함
  - 랭킹 하락이 콴다 단독 문제인지 전체 키워드 시장 변동인지 즉시 구분 불가

opportunities:
  - Slack 알림에 해당 키워드 대시보드 딥링크 포함
  - 키워드 상세 뷰에서 랭킹 추이 + CVR + 다운로드 + 경쟁사 비교를 단일 화면에 통합
  - 경쟁사 동일 키워드 랭킹을 같은 차트에 오버레이로 표시
```

### Journey J-04: PM 성과 자가 조회 (P-03 재현)

```
journey_id   : J-04
persona      : P-03 (재현, PM)
goal         : 담당자에게 요청하지 않고 이번 주 ASO 성과 현황 파악
trigger      : 주간 팀 미팅 전 성과 확인 필요 또는 경영진 보고 준비
priority     : Medium (P1 Feature - Weekly Summary)

touchpoints:
  1. [진입] 대시보드 접속 → 요약(Summary) 뷰로 기본 랜딩
  2. [지표 확인] 주력 키워드 평균 랭킹, 전주 대비 변화, ASO 점수 현황 확인
  3. [다운로드 추이] 이번 주 다운로드/CVR 추이 차트 확인
  4. [특이사항 파악] 하이라이트된 주요 변동 사항 요약 카드 확인
  5. [리포트 공유] 요약 화면을 PDF 또는 Slack으로 공유

emotional_arc:
  진입(자율성 기대) → 지표 확인(몰입) → 긍정 지표(만족)/부정 지표(우려)
  → 특이사항 파악(맥락 이해) → 리포트 공유(완료)

key_pain_points:

### REQ-P1-02: 랭킹 변동 하이라이트 (녹/적 색상)

```
requirement_id   : REQ-P1-02
requirement_text : 전일 대비 랭킹 상승/하락을 색상(녹/적)으로 자동 표시

ui_implications:
  - 색상만으로 상태를 전달하면 접근성 위반 (WCAG 1.4.1 Use of Color)
  - 색상 + 방향 아이콘(↑↓) + 숫자를 함께 사용해야 함
  - 변동 없음(→)도 명시적으로 표시하여 "데이터 없음"과 구분

component_needs:
  - 변동 배지 컴포넌트 (재사용 가능)
    - 상승: ↑N (녹색 배경 또는 녹색 텍스트 + 상향 화살표 아이콘)
    - 하락: ↓N (빨간색 배경 또는 빨간색 텍스트 + 하향 화살표 아이콘)
    - 유지: → (회색, 변동 없음)
    - 데이터 없음: — (회색, 전일 데이터 부재)
  - 색맹 접근성 고려: 색상 외 패턴/모양 보조 수단 고려
    - 대안: 테두리 두께, 아이콘 모양으로도 구분 가능하도록

state_requirements:
  - 각 변동 상태가 색상+아이콘+숫자로 명확히 구분됨
  - 툴팁: 해당 배지 hover 시 "전일(YYYY-MM-DD) 대비 N위 상승/하락" 설명

accessibility_note:
  WCAG 2.1 기준 1.4.1 (Use of Color): 색상만으로 정보 전달하지 않을 것.
  ↑↓ 아이콘과 숫자 조합 필수. 색맹 사용자를 위한 대비 충분 여부 확인 필요.
```

  - 경쟁사 데이터는 센서타워 추정치이며 실제 값과 다를 수 있으므로 출처 명시

  - 경쟁사 비교는 통합 뷰의 확장 컬럼 또는 별도 비교 탭으로 구성

component_needs:
  - 경쟁사 비교 컬럼 그룹 (토글 가능)
    - 기본 표시: 콴다 랭킹 + 가장 강한 경쟁사 1-2개
    - 확장 시: 전체 경쟁사 4개 랭킹 컬럼
  - 경쟁사 선택 체크박스 (어느 경쟁사를 표시할지 선택)
  - 경쟁사별 색상 구분 (콴다=파랑, 말해보카=노랑 등)
  - 데이터 출처 레이블 ("센서타워 추정치 기준")
  - 경쟁사 순위가 없는 경우 "—" 표시

state_requirements:
  - success  : 경쟁사 랭킹 데이터 표시
  - no_data  : 해당 키워드에서 경쟁사 랭킹 없음 - "미순위" 또는 "100위 밖"
  - loading  : 경쟁사 데이터 로딩 중 - 컬럼 스켈레톤
```

## Section 4. Architecture-Driven UX Constraints

2-tier 아키텍처(Google Sheets + 웹 대시보드)에서 비롯되는 UX 설계 제약:

### C-01: 데이터 신선도 표시 의무

Google Apps Script는 6분 실행 시간 제한이 있어 수집이 분할 실행될 수 있다. 모든 화면에서 마지막 성공 수집 시각을 항상 표시해야 한다. "오늘 데이터"와 "어제 데이터"를 사용자가 구분할 수 있어야 한다.

### C-02: Android 데이터 수동 입력 경로

Google Play Console 키워드 API 불가로 Android 키워드 성과는 수동 입력이다. 수동 입력 데이터 컬럼에는 "수동 입력 (주 1회)" 레이블과 마지막 업데이트 날짜를 표시하여 자동 수집 데이터와 명확히 구분해야 한다.

### C-03: 센서타워 API 불확실성 대비

센서타워 키워드 API 엔드포인트 접근 불가 시 CSV Export 우회 가능성이 있다. UI는 데이터 수집 방식에 무관하게 동일하게 작동해야 하며, 수집 경로 변경 시 사용자에게 영향이 없어야 한다.

### C-04: 소규모 팀 내부 도구 특성

사용자가 3명 이하이므로 복잡한 권한 관리보다 신뢰 기반 단순 접근이 적합하다. 단, API 키/인증 정보는 UI에 절대 노출되지 않아야 한다.

## Section 6. Design Recommendations (우선순위 순)

**Priority 1 - Critical (P0 기능 구현 시 반드시 적용)**

1. 모든 화면 상단 고정 헤더에 데이터 수집 상태 표시 컴포넌트 배치. "마지막 업데이트 시각 + 성공/실패 상태"가 항상 가시적이어야 한다.

2. 통합 뷰 테이블의 정보 계층 설계: 키워드명(1열, 고정), 분류 배지, 현재랭킹, 전일대비 변동(색상+아이콘+숫자), 7일 스파크라인, 지표 컬럼 순으로 좌에서 우로 배열. 가장 중요한 정보가 스크롤 없이 보여야 한다.

3. 주력/보조 탭 분류를 최상위 내비게이션에 배치. 이 분류가 모든 뷰의 1차 필터이므로 진입점에서 즉시 접근 가능해야 한다.

**Priority 2 - High (P0-P1 전환 시 적용)**

4. 변동 알림 딥링크 설계: Slack 알림 → 대시보드 해당 키워드 상세 뷰 직접 연결. 알림 수신 후 3클릭 이내에 원인 분석이 완료되어야 한다.

5. PM(재현) 전용 요약 뷰 설계: 상세 테이블 없이 KPI 카드 + 주간 요약만 표시하는 별도 뷰. 재현이 대시보드에 진입했을 때 기본 랜딩 페이지로 설정 가능해야 한다.

**Priority 3 - Recommended (안정화 단계 적용)**

6. 키워드 제안 채택 플로우 최적화: 제안 목록 → 채택 버튼 → 분류 선택 → 완료 토스트까지 단일 모달로 3단계 완료.

7. Android/iOS 데이터 출처 레이블 일관성: 모든 지표 컬럼 헤더에 플랫폼 아이콘 배지 적용하여 데이터 출처 모호함 제거.

8. 빈 상태(Empty State) 디자인: 각 탭/섹션에서 데이터가 없을 때 단순 빈 화면이 아닌, 다음 액션(키워드 추가, 필터 변경 등) CTA를 명확히 제시.
